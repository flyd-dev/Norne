# Norne — Intern Chatbot (MVP)

An internal company chatbot. Users ask questions in natural language and get
answers generated by a **pluggable LLM** (OpenAI or a local Ollama/Llama model),
grounded in data read from **Firestore**
(`accounts`, `projects`, `projects/{id}/budget_lines`, `projects/{id}/quantities`).

All Firestore and LLM access happens **server-side only**. No secrets ever
reach the browser.

---

## Tech stack

- **Next.js (App Router)** + **TypeScript**
- **Firebase Admin SDK** (preferred) or **Firestore REST API** (fallback)
- **Pluggable LLM provider**: **OpenAI** or **Ollama** (local Llama)
- **Document knowledge base**: upload PDF/DOCX/PPTX/TXT/CSV/XLSX/MSG → hybrid retrieval (semantic vector search via embeddings, with keyword fallback), stored in a local JSON file + a local `sqlite-vec` index (not Firestore)
- **SharePoint sync** (optional): pull a SharePoint document library into the knowledge base via Microsoft Graph (app-only auth), resumable + incremental
- Server-side route at **`/api/chat`** + admin routes at **`/api/admin/documents`**
- Simple React chat UI + protected admin upload page at **`/admin/documents`**

### Added dependencies

| Package | Why |
|---|---|
| `xlsx` (SheetJS, from the SheetJS CDN) | Parse XLSX spreadsheets (per-sheet text). Installed from `https://cdn.sheetjs.com/...` because the npm build has an unpatched high-severity advisory. |
| `mammoth` | Extract raw text from DOCX files. |
| `pdf-parse` | Extract text from PDF files (imported via its inner `lib/pdf-parse.js`). |
| `adm-zip` | Read PPTX slide/notes XML (a .pptx is a zip) to extract text. |
| `@kenjiuno/msgreader` | Extract subject/sender/body text from Outlook `.msg` files. |

The orchestrator depends only on the `LLMProvider` interface (swap via
`LLM_PROVIDER`) and on `searchDocuments(query)` — which now runs semantic vector

falling back to keyword search otherwise. Neither requires orchestrator changes.

### Semantic search & SharePoint sync

- **Embeddings** are pluggable via `EMBEDDINGS_PROVIDER`. Anthropic/Claude has no
  embeddings API, so this is a separate provider (Claude still writes the
  answers): `voyage` (Anthropic's recommended provider — hosted, free tier),
  `ollama` (free + local), `openai`, or `none`. Vectors live in a local
  `sqlite-vec` file (`VECTOR_STORE_PATH`). Uploads auto-index; backfill existing
  docs with `node scripts/reindex-documents.mjs`.
- **SharePoint sync** (set `SHAREPOINT_ENABLED=true` + `SHAREPOINT_*`) pulls a
  document library via Microsoft Graph. It is resumable and incremental (per-drive
  delta cursors), so a large library loads over several batches and later runs
  only pick up changes. Drive it with `node scripts/sync-sharepoint.mjs`. No paid
  Azure subscription — the app registration lives in Entra ID, included with the
  Microsoft 365 tenant that hosts SharePoint.

## Firestore access: Admin SDK mode vs REST fallback mode

The app validates its configuration on the first request (`validateEnv()` in
`lib/env.ts`) and picks a backend automatically based on which variables are set:

| Mode | When used | Needs |
|---|---|---|
| **Admin SDK mode** (preferred) | A service-account key is present | `FIREBASE_CLIENT_EMAIL` + `FIREBASE_PRIVATE_KEY` |
| **REST fallback mode** | No service-account key, only web API key + auth user | `FIREBASE_API_KEY` + `FIREBASE_AUTH_EMAIL` + `FIREBASE_AUTH_PASSWORD` |

If **both** are set, Admin SDK mode wins. If **neither** is fully set, the first
request fails fast with a clear error (variable names only, never values).

### Admin SDK mode (recommended)

The Firebase Admin SDK uses a **service-account private key**, authenticates as
the project itself, and is the proper way for a trusted server to read Firestore.
Generate a key in **Firebase Console → Project settings → Service accounts →
Generate new private key**, then set `FIREBASE_CLIENT_EMAIL` and
`FIREBASE_PRIVATE_KEY` (keep the `\n` escapes, wrap the value in quotes).

### REST fallback mode (temporary, less ideal)

If you only have a **web API key + an email/password Firebase Auth user** (the
flow documented in `api-service-account.md`), the Admin SDK cannot be used, so
the app signs in as that user and calls the Firestore REST API.

> ⚠️ **REST mode is a temporary stopgap and less ideal.** It stores a real user
> password in env vars, authenticates as a regular user (so reads are bound by
> that user's security rules), and depends on token refresh. **Migrate to Admin
> SDK mode as soon as a service-account key is available**, and rotate any
> password/API key that has been shared in plaintext.

## LLM provider: OpenAI vs Ollama (local Llama)

The model is pluggable. Choose it with `LLM_PROVIDER` (default `openai`). The
orchestrator only depends on the `LLMProvider` interface, so switching providers
needs no code change.

### OpenAI mode (default)

```bash
LLM_PROVIDER=openai
OPENAI_API_KEY=sk-...
OPENAI_MODEL=gpt-4o-mini   # optional
```

Hosted API; nothing to run locally. Requires an API key and bills per call.

### Ollama / local Llama mode

Run a model yourself — no API key (for local), no per-call cost.

1. **Install Ollama** — see <https://ollama.com/download> (macOS/Linux/Windows).
2. **Pull a model:**
   ```bash
   ollama pull llama3.1
   ```
   (Ollama serves its HTTP API on `http://localhost:11434` by default.)
3. **Configure the app:**
   ```bash
   LLM_PROVIDER=ollama
   OLLAMA_MODEL=llama3.1
   OLLAMA_BASE_URL=http://localhost:11434   # optional; this is the default
   OLLAMA_API_KEY=                          # optional; see remote mode below
   ```
   No `OPENAI_API_KEY` is required in this mode.

#### Deployment modes

**Same server as the app** (Ollama runs on localhost next to Next.js):

```bash
OLLAMA_BASE_URL=http://localhost:11434
```

**Separate VPS / external endpoint** (Ollama runs on its own machine):

```bash
OLLAMA_BASE_URL=https://your-ollama-domain.example.com
OLLAMA_API_KEY=your-reverse-proxy-bearer-token   # if the endpoint requires auth
```

When `OLLAMA_API_KEY` is set, the app sends `Authorization: Bearer <key>` with
every request. It is **optional** — leave it blank for local Ollama, which
normally has no authentication.

> ⚠️ **Security — do not expose Ollama directly to the public internet
> unprotected.** Ollama itself has no built-in authentication. If you run it on a
> remote VPS, protect it with at least one of:
> - **firewall rules** (allow only the app server's IP),
> - a **VPN / private network** between the app and Ollama,
> - a **reverse proxy with authentication** (e.g. nginx/Caddy requiring a bearer
>   token) — in which case set `OLLAMA_API_KEY` to that token.
>
> An open Ollama endpoint lets anyone run models on (and read traffic to) your
> server.

### Tradeoffs

| | OpenAI | Ollama (local Llama) |
|---|---|---|
| Setup | Easiest (just a key) | Install Ollama + pull a model |
| Quality | Usually stronger | Can be weaker, model-dependent |
| Speed | Fast, hosted | Depends on your hardware; can be slower |
| Cost | Per-call billing | No per-call cost |
| Privacy | Data sent to OpenAI | Stays local/private |
| Resources | None local | Needs local/server CPU/GPU + RAM |

Pick OpenAI for the easiest, generally strongest results; pick Ollama when you
need data to stay local/private or want to avoid per-call cost, accepting the
local resource requirements and potentially weaker/slower answers.

## Deploying on a VPS (Ollama on the same server)

This deploys the whole app on one VPS, with Ollama running locally on that same
machine. The Next.js server calls Ollama at `http://localhost:11434` — **no SSH
tunnel, no public exposure of Ollama**.

> Tested target: host `srv1695266.hstgr.cloud`, model `llama3.1:8b`, CPU-only.
> CPU inference is noticeably slower than a hosted API or GPU — expect multi-second
> responses, especially for longer answers.

### 1. Install Ollama on the VPS

```bash
curl -fsSL https://ollama.com/install.sh | sh
sudo systemctl enable --now ollama        # runs Ollama as a service on :11434
```

By default Ollama binds to `127.0.0.1:11434` (localhost only) — keep it that way.
Do **not** set `OLLAMA_HOST=0.0.0.0`.

### 2. Pull the model

```bash
ollama pull llama3.1:8b
ollama run llama3.1:8b "hei"   # quick smoke test (Ctrl-D to exit)
```

### 3. Keep port 11434 closed to the public internet

Ollama has no authentication. Since the app talks to it over localhost, the port
never needs to be open externally. Verify your firewall blocks it (example: UFW):

```bash
sudo ufw deny 11434/tcp        # belt-and-suspenders; localhost binding already protects it
sudo ufw status
```

(If you instead run Ollama on a *separate* machine, see *LLM provider → Ollama
mode → Separate VPS* above and protect it with a reverse proxy + `OLLAMA_API_KEY`.)

### 4. Get the app onto the VPS and configure it

```bash
git clone <your-repo-url> /var/www/norne-chatbot   # or copy the files up
cd /var/www/norne-chatbot
npm ci                                              # install exact deps

cp .env.production.example .env.local               # then edit .env.local
```

Set these in `.env.local` (this file is git-ignored — **never commit it**):

```bash
LLM_PROVIDER=ollama
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=llama3.1:8b
# OLLAMA_API_KEY=            # leave blank: local Ollama needs no auth

FIREBASE_PROJECT_ID=...
# plus ONE Firestore backend (Admin SDK preferred — see Setup above)
FIREBASE_CLIENT_EMAIL=...
FIREBASE_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----\n"
# No OPENAI_API_KEY needed in Ollama mode.
```

### 5. Build and run

```bash
npm run build
npm run start            # serves on http://0.0.0.0:3000 by default
```

Set a port with `PORT=3000 npm run start`. Put the app behind a reverse proxy
(nginx/Caddy) for TLS on port 443; only the app needs to be public, not Ollama.

### 6. Keep it running (PM2 or systemd)

Both load env from `.env.local` / `.env.production` in the working directory, so
**no secrets live in the process manager config**.

**PM2:**

```bash
npm i -g pm2
pm2 start deploy/ecosystem.config.cjs
pm2 save
pm2 startup            # follow the printed command to start PM2 on boot
```

**systemd:** copy and edit the sample unit, then:

```bash
sudo cp deploy/norne-chatbot.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now norne-chatbot
sudo journalctl -u norne-chatbot -f     # logs
```

Edit `WorkingDirectory`/`cwd`, `User`, and `PORT` in those files to match your VPS.

## Document knowledge base (upload + search)

The bot can also answer from uploaded company documents (e.g. an Excel
*bemanningsplan*), alongside the Firestore project data.

### Uploading documents

1. Set `ADMIN_UPLOAD_TOKEN` to a long random value in `.env.local` (server-side
   only — never committed, never exposed to the browser).
2. Open **`/admin/documents`**, enter the token to unlock, then upload files.
3. Each upload shows name, type, upload date and number of chunks, with a delete
   button. The token is held only in the browser session (`sessionStorage`).

### Supported file types

**PDF, DOCX, PPTX, TXT, CSV, XLSX, MSG.** Max **10 MB** per file; other types are rejected.
For XLSX, each sheet is extracted separately, preserving the sheet name, headers
and row values (good for staffing plans / bemanningsplaner). Only the **extracted
text/chunks are stored** — the original file is never persisted.

### Storage (local file, not Firestore)

Uploaded-document metadata and chunks are stored in a **single local JSON file**
on the server — **not** in Firestore. Firestore is used only for project data.
The path is configurable via `DOCUMENT_STORE_PATH` (default
`/var/lib/norne-chatbot/knowledge-documents.json`); the directory is created
automatically, and the original uploaded file is never persisted.

```jsonc
// DOCUMENT_STORE_PATH
{
  "documents": [
    {
      "id": "…", "name": "bemanningsplan.xlsx", "fileType": "xlsx",
      "uploadedAt": "2026-01-01T00:00:00Z", "chunkCount": 12,
      "chunks": [
        { "documentId": "…", "documentName": "bemanningsplan.xlsx",
          "fileType": "xlsx", "sheetName": "Bemanning", "chunkIndex": 0,
          "text": "…", "uploadedAt": "2026-01-01T00:00:00Z" }
      ]
    }
  ]
}
```

Text is split into ~800–1200 char chunks with ~150–250 char overlap.

> **Why local file?** For this demo, Firestore security rules block the new
> document collection. Local JSON storage avoids that entirely. Trade-offs: the
> store lives on one server (not shared across instances) and is not backed up by
> Firestore — fine for a single-VPS demo. To productionize, move it to Firestore
> (Admin SDK) or object storage and reimplement `lib/documents/store.ts`.

### How search works (current limitations)

- Every chat question runs a **keyword / term-frequency search** over stored
  chunks (`lib/rag/documentSearch.ts`) and the top ~6 chunks are added to the
  model context. Document names appear in `sources`, and `dataUsed.documents`
  lists the chunk references used (no raw chunk text is returned to the client).
- **This MVP uses keyword search, not embeddings.** It does not understand
  synonyms or meaning, ranks purely on term overlap, and reads all chunks per
  query (fine for modest volumes, not for very large corpora).

### Upgrading to vector search later

`searchDocuments(query)` is the only contract the orchestrator depends on. To
upgrade: compute embeddings for each chunk at upload time (store the vector on
the chunk), then reimplement `searchDocuments` to embed the query and run a
vector similarity search — no orchestrator or API changes required.

## API

### `POST /api/chat`

Request:

```json
{ "message": "Vis budsjettlinjer for prosjekt GSLeXiSkaiAkEqcuFxIx" }
```

Response:

```json
{
  "answer": "…",
  "sources": ["projects", "bemanningsplan.xlsx"],
  "dataUsed": {
    "firestoreCollections": ["projects"],
    "documents": [
      {
        "documentId": "…",
        "documentName": "bemanningsplan.xlsx",
        "fileType": "xlsx",
        "sheetName": "Bemanning",
        "chunkIndex": 2
      }
    ]
  },
  "warnings": []
}
```

- `answer` — natural-language answer (Norwegian by default)
- `sources` — all sources the answer draws on (Firestore collection paths plus
  any document names used)
- `dataUsed.firestoreCollections` — Firestore collection paths read for this answer
- `dataUsed.documents` — references to document chunks used (no chunk text)
- `warnings` — non-fatal notices, e.g. truncation ("showing 50 of 320 accounts"),
  an ambiguous/missing project, or large row sets aggregated before sending

Error responses use `{ "error": string, "requestId": string }`. The `error` is
always generic; the `requestId` lets you find the matching server log line.

## Endre API integration (optional, disabled by default)

The app can *optionally* talk to [Endre](https://public-api.endre.app)'s public
REST API to read live project/economy data. This is an **optional integration
layer** — it is **not** wired into `/api/chat` answers, and the existing
Firebase + uploaded-document flow remains the source of truth. With the
integration off (the default), nothing changes and no Endre credentials are
required to start the app.

### How it works

- **Auth:** OAuth2 *password* grant. The client posts `username`/`password` (and
  optional `client_id`/`client_secret`) to `POST /token`, receives a JWT bearer
  token, caches it until expiry (read from the JWT `exp` claim, with a
  conservative TTL fallback), and re-authenticates automatically when it
  expires or a request returns `401`.
- **Safety:** passwords and tokens are never logged or placed in error messages;
  every request has a timeout; failures throw typed errors
  (`EndreConfigError`, `EndreAuthError`, `EndreApiError`, `EndreTimeoutError`).
- **Read-only helpers** are provided only for endpoints that actually exist in
  the Endre OpenAPI spec: `listOrganizations`, `getOrganization`,
  `listProjects`, `getProject`, `getProjectAmounts`, `listProjectCases`,
  `listProjectContracts`, `getProjectTags`, `listProjectOrganizations`
  (see [`lib/endre/client.ts`](lib/endre/client.ts)).

> **Note on requested helpers:** the Endre API has **no** chart-of-accounts,
> budget-line, quantity, or staffing/capacity/timer endpoints. So
> `listAccounts()`, `getBudgetLines()`, `getQuantities()`, and
> `getStaffingCapacity()` are intentionally **not** implemented — the nearest
> real data is project *amounts*, *cases*, *cost-items*, and *contracts*.

### Environment variables

```bash
ENDRE_API_ENABLED=false                       # master flag; default false
ENDRE_API_BASE_URL=https://public-api.endre.app
ENDRE_API_USERNAME=                           # required to use the integration
ENDRE_API_PASSWORD=                           # required to use the integration
ENDRE_API_CLIENT_ID=                          # optional (only if Endre issues one)
ENDRE_API_CLIENT_SECRET=                      # optional
```

The integration is used **only when `ENDRE_API_ENABLED=true` *and* a username +
password are present.** Otherwise `getEndreClient()` returns `null` and callers
fall back to the existing Firebase/document flow.

### Enabling it

1. Put real credentials in `.env.local` (never commit them).
2. Set `ENDRE_API_ENABLED=true`.
3. Verify connectivity via the diagnostic endpoint below.

### Diagnostic endpoint

`GET /api/admin/endre/status` — protected by `ADMIN_UPLOAD_TOKEN` (same bearer
gate as the other admin routes). It returns, without exposing any credentials,
tokens, or raw payloads:

```json
{
  "enabled": true,
  "configured": true,
  "canAuthenticate": true,
  "availableCapabilities": ["organizations", "projects", "project_amounts", "..."],
  "error": "…only present, and generic, if authentication failed"
}
```

```bash
curl -H "Authorization: Bearer $ADMIN_UPLOAD_TOKEN" \
  http://localhost:3000/api/admin/endre/status
```

## Configuring project name fields

Questions about budget lines or quantities need a specific project. The resolver
(`lib/chat/projectResolver.ts`) matches a project by its document id **or** by a
human-readable name. Because the exact Firestore field is project-specific, the
name fields are configurable:

```ts
// lib/chat/projectResolver.ts
export const PROJECT_NAME_FIELDS = [
  "project_name",   // verified live schema: full human-readable name
  "project_number", // verified live schema: short 4-char project number
  "name",
  "navn",
  "title",
  "projectName",
  "displayName",
] as const;
```

The first two match the live `projects` schema (verified read-only); the rest are
generic fallbacks. Edit this list to match your schema (priority order). The
resolver returns a clear message when **no** project matches and when **multiple**
projects match the same name (asking the user to disambiguate by id).

## Assistant behaviour

- Answers in **Norwegian** by default (switches if the user clearly uses another language).
- Concise but useful.
- **Only** answers from the retrieved context — does not invent facts.
- If the data isn't available, it says so clearly.
- If project-specific data is requested without a project id/name, it asks the
  user to specify the project and lists available projects.
- Mentions the data source where possible.

## Known limitations

- **Read-only.** The bot can read Firestore data but never writes.
- **Heuristic intent detection.** Topic routing uses keyword matching
  (Norwegian + English); unusual phrasings may route imperfectly.
- **Heuristic project-name matching.** Name resolution is substring-based over
  `PROJECT_NAME_FIELDS`. It prefers the longest match to disambiguate overlaps,
  but unusual naming can still mis-match — prefer passing a project id.
- **Data is minimized before the model sees it.** Top-level collections are
  capped (max 50 records each), documents are reduced to compact scalar fields
  (long strings truncated, nested objects dropped), and budget lines / quantities
  are **aggregated** (count + numeric totals + a small sample) rather than sent in
  full. This bounds token cost but means the model reasons over summaries, not
  every raw row. Truncation/aggregation is surfaced in `warnings`.
- **REST fallback mode is temporary** (see above) — migrate to Admin SDK mode.
- **No authentication / per-user access control** on the chat endpoint yet — it
  is intended for internal/network-restricted use during the MVP. Add auth
  before any wider exposure.
- **No conversation memory.** Each request is answered independently (no history
  is sent back to the model).
- **Single LLM call per question.** No tool-calling / multi-step reasoning.
- **Document search is keyword-based, not embeddings.** See *Document knowledge
  base* above — no semantic/synonym matching yet, and all chunks are scanned per
  query (fine for modest volumes).
- **Admin upload uses a single shared token.** No per-user admin accounts/roles.
- **Uploaded documents live in a local JSON file on one server** (not Firestore,
  not shared across instances, not backed up). Fine for a single-VPS demo; move
  to a shared store to productionize.
